## 결정 사항

### [2026.03.24] Supabase Client 주입 방식
- **맥락**: NestJS에서 Supabase Client를 각 모듈에서 어떻게 접근할지 결정 필요
- **결정**: 글로벌 모듈(SupabaseModule)로 만들어서 SupabaseService를 통해 주입
- **이유**: 모든 모듈이 동일한 클라이언트 인스턴스를 사용하도록 일관성 보장. ConfigService 의존성도 한 곳에서 처리.
- **대안**: 각 모듈에서 직접 createClient 호출 (중복 코드, 설정 불일치 위험)

### [2026.03.24] 카테고리 매핑 규칙 저장소
- **맥락**: 거래 내역의 가맹점명 → 카테고리 자동 분류 규칙을 어디에 저장할지
- **결정**: MVP에서는 JSON 상수 파일(src/codef/constants/category-rules.json)로 관리. 추후 DB 테이블로 이전 가능.
- **이유**: MVP 단계에서 규칙이 자주 바뀌지 않고, 배포 시 코드와 함께 반영되는 게 더 간단.
- **대안**: DB 테이블 (관리자 UI 필요, MVP에서는 과도)

### [2026.03.24] 스크랩 AI 요약 실패 처리
- **맥락**: 유튜브 자막 없음, 파싱 불가 사이트, AI API 오류 등 요약 실패 시 처리 방법
- **결정**: summary_status 필드를 두고 'pending' | 'completed' | 'failed' 상태 관리. 실패 시 프론트에 "요약 준비 중" 표시. 최대 3회 재시도 후 'failed' 확정.
- **이유**: 스크랩 저장 자체는 요약 성공 여부와 무관하게 즉시 완료되어야 함. 요약은 비동기 후처리.
- **대안**: 요약 실패 시 스크랩 거부 (UX 저하)

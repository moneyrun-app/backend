## [2026-04-02] MVP v2.0 — 3페이지로 대폭 축소

**결정:** 5탭 풀 기능 → 3페이지(페이스메이커/마이북/마이페이지) 집중 MVP

**이유:**
- MVP는 핵심 가치 검증이 목적. 페이스메이커(AI 일일 코칭) + 마이북(콘텐츠 모음)이 핵심
- 커뮤니티, 자산 트래킹, 시뮬레이션 등은 유저 확보 후 의미
- 빠른 출시 > 완벽한 기능

**제거된 기능:** 커뮤니티, 자산 트래킹, 65세 시뮬레이션 독립 페이지, AI PDF 이메일 발송, 주간 체크인 UI, 소득 구간

**영향 범위:** 전체 문서 v1.0 → v2.0 재작성

## [2026-04-02] 온보딩 간소화 — 7문항 → 4스텝

**결정:** 진단 7문항을 4스텝 온보딩(나이, 실수령액, 좋은 소비, 고정 소비)으로 축소

**이유:**
- 기존 7문항 중 저축 방식·수익률·비상금·소비 패턴은 MVP에서 불필요
- 핵심은 "잉여자금 = 하루 쓸 수 있는 돈" 계산에 필요한 최소 데이터
- 비로그인 진단 제거 → 로그인 후 바로 온보딩

**제거된 입력:** 저축 방식(복수선택), 예상 수익률, 비상금 유무, 소비 패턴

**영향 범위:** onboarding/ 모듈, 진단 관련 코드 전체 제거

## [2026-04-02] 등급 판정 기준 변경 — 연금 예상액 → 저축 비율

**결정:** 65세 연금 시뮬레이션 대신 좋은 소비(저축) 비율로 등급 판정

**이유:**
- 시뮬레이션 독립 페이지를 제거했으므로 복잡한 연금 계산이 등급 판정의 핵심이 아님
- "실수령액 대비 저축 비율"이 더 직관적이고 행동 변화에 가까움

**새 기준:**
- RED: 저축 0원 또는 비율 < 10%
- YELLOW: 10~20%
- GREEN: ≥ 20%

**영향 범위:** grade 계산 로직, pension.calculator.ts 제거

**영향 범위:** pacemaker/ 모듈, pacemaker_messages 테이블

## [2026-04-02] 잉여자금 역산 방식 채택

**결정:** 실수령액에서 좋은 소비·고정 소비를 빼서 잉여자금(하루 사용 가능 금액)을 역산

**이유:**
- "하루에 얼마 쓸 수 있어"가 가장 직관적인 핵심 지표
- 유저 입장: 복잡한 시뮬레이션보다 "오늘 얼마까지 쓸 수 있는지"가 더 중요

**공식:** 잉여자금(월) = 실수령액 - Σ(좋은소비) - Σ(고정소비), 일 = 월 ÷ 30

**영향 범위:** surplus.calculator.ts (백엔드), lib/surplus.ts (프론트)
